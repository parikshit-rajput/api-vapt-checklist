Overview 4 items

  • Use this checklist to test REST/JSON APIs, mobile-backed APIs, web APIs, GraphQL endpoints, webhooks, callbacks, and third-party API integrations.

  • Capture the API request, perform the checklist item, compare responses across users/roles, and mark the item as checked only after validating the secure behavior.

  • At least two normal users and one higher-privilege/admin test account where possible.

  • Burp Suite, Postman, curl, browser DevTools, jwt.io, Burp Collaborator/interactsh, Turbo Intruder, ffuf/dirsearch, Swagger/OpenAPI tools.

API Testing Setup & Inventory 10 items

  • Use Burp Proxy, browser DevTools, mobile proxy, or Postman to capture every API request generated during normal app usage.

  • Use at least two normal users and one higher-privilege/admin test account, if available. Keep separate tokens/cookies for each account.

  • Record API domains and versions such as /api, /api/v1, /api/v2, /rest, /graphql, /mobile, /admin, /internal.

  • Identify whether APIs use Bearer JWT, session cookie, API key, OAuth token, mTLS, OTP, SSO, or custom headers.

  • For each API, note method, path, parameters, headers, body, auth requirement, role requirement, response fields, and status codes.

  • Check /swagger, /swagger-ui, /openapi.json, /api-docs, /v2/api-docs, /graphql, and documentation links exposed in JS files.

  • Search Burp history and JS files for /v1, /v2, /beta, /legacy, /test, /staging, /mobile, /admin, /internal endpoints.

  • Group endpoints into auth, user/profile, admin, upload, search, payment, notification, webhook, report/export, and third-party callback flows.

  • Record whether APIs accept JSON, form-urlencoded, multipart, XML, GraphQL, protobuf, or raw body formats.

  • Call each sensitive endpoint without token, with invalid token, and with low-privilege token to confirm access is blocked.

API1: Broken Object Level Authorization (BOLA / IDOR) 10 items

  • Use User A token and change /users/A/profile to /users/B/profile or /orders/1001 to /orders/1002. User A must not access User B's object.

  • Modify values like ?user_id=, ?order_id=, ?file_id=, ?invoice_id=, ?post_id=, ?transaction_id= and verify object-level authorization.

  • Modify JSON fields such as userId, doctorId, organizationId, tenantId, accountId, fileId, or ownerId and replay the request.

  • Check headers such as X-User-ID, X-Account-ID, X-Tenant-ID, X-Org-ID, X-Role-ID. Backend must not trust user-controlled headers.

  • Modify nested values like { user: { id: 123 } }, { owner: { id: 123 } }, { tenant: { id: 123 } } and verify access checks.

  • Decode and modify base64, URL-encoded, UUID, numeric, or hash-like IDs. Example: base64 user_id values in parameters or body.

  • Check whether list APIs return only current user's objects. Try filters like user_id, owner_id, tenant_id, status, createdBy.

  • Repeat BOLA checks on PUT, PATCH, and DELETE APIs. Many apps protect GET but miss update/delete authorization.

  • If app is multi-tenant, use Tenant A token and attempt to access Tenant B resources by changing tenant/org identifiers.

  • Change file IDs, document IDs, signed URL IDs, attachment IDs, or media IDs to access another user's uploaded files.

API2: Broken Authentication 11 items

  • Remove Authorization header/cookie and verify protected APIs return 401/403, not user data.

  • Replace token with random text, truncated token, expired token, or different token type and confirm rejection.

  • Login, capture token, logout, then replay the old token. It should be invalid or blocked as per policy.

  • Reset password and replay previous access/refresh tokens. Old sessions/tokens should be revoked or re-auth required.

  • Check refresh token expiry, reuse, rotation, replay, and whether refresh token can generate unlimited access tokens.

  • Decode JWT, change user_id/role/isAdmin/tenant_id/exp and replay. API must reject modified tokens.

  • Try alg=none or weak/incorrect signing only in authorized testing. API must validate algorithm and signature strictly.

  • Verify exp, nbf, iss, aud, tenant, and user claims are validated correctly.

  • Inspect URLs, responses, errors, logs, browser storage, Referer headers, and analytics calls for access/refresh tokens.

  • Attempt repeated login attempts for same user/IP and verify lockout, throttling, CAPTCHA, or abuse protection.

  • Try repeated OTP guesses, OTP reuse, expired OTP, null/000000 values, and direct access after only first-factor authentication.

API3: Broken Object Property Level Authorization (BOPLA) 8 items

  • Review responses for unnecessary fields such as internal IDs, email, phone, address, DOB, role, permissions, tokens, flags, or debug fields.

  • Call the same API as guest, normal user, premium user, and admin. Low-privilege users should not receive admin/internal properties.

  • Add hidden fields like isAdmin, role, status, verified, plan, price, balance, permissions, organizationId, tenantId in request body.

  • Add nested objects such as { profile: { role: 'admin' } } or { permissions: ['admin'] } and verify backend ignores/rejects unauthorized fields.

  • Use PATCH/PUT to update fields not available in UI, such as emailVerified, phoneVerified, accountStatus, userType, creditBalance.

  • Modify IDs, createdBy, ownerId, createdAt, approvedBy, paymentStatus, or moderationStatus. Server must not accept client-controlled system fields.

  • Admin-only fields should not be visible or modifiable by normal users, even if the endpoint itself is accessible.

  • Inspect arrays like permissions, roles, teams, groups, subscriptions, invoices, and linked accounts for excessive data or update abuse.

API4: Unrestricted Resource Consumption 9 items

  • Send repeated requests to login, forgot password, OTP send, OTP verify, resend OTP, and refresh-token endpoints.

  • Repeat search, filter, report, export, upload, download, PDF generation, and AI/content generation APIs.

  • Send very large JSON/form/multipart payloads and confirm size limits, timeout handling, and no server crash.

  • Modify limit/pageSize/per_page to large values such as 10000 or 999999. API should enforce max limits.

  • Send complex filters, nested filters, broad date ranges, multiple sort fields, or regex-like searches and observe performance/limits.

  • Upload files above allowed size and verify rejection before heavy processing/storage.

  • Send many parallel requests to expensive endpoints and verify throttling/queuing/limits.

  • Try bypassing limits by changing IP headers, user-agent, device ID, email case, phone format, or API version.

  • Trigger failed uploads/exports/jobs and verify temporary files/jobs do not accumulate or remain accessible.

API5: Broken Function Level Authorization (BFLA) 7 items

  • Use a normal-user token to call /admin, /api/admin, /management, /internal, /moderation, /reports, or role-management APIs.

  • Call APIs not visible in UI but present in JS/Burp history, such as delete user, approve request, export data, update role.

  • Try GET/POST/PUT/PATCH/DELETE on the same admin or sensitive endpoint. Authorization must apply to every method.

  • Modify role, permission, isAdmin, userType, accountType, plan, or status fields in API requests.

  • Check whether /mobile, /internal, /legacy, /v1, or admin APIs enforce the same permissions as current web APIs.

  • Call bulk-delete, bulk-update, import, export, approve-all, or status-change APIs as low-privilege users.

  • Try approving, rejecting, publishing, disabling, enabling, or deleting content/users without the required role.

API6: Unrestricted Access to Sensitive Business Flows 8 items

  • Automate resend OTP, forgot password, invitation, verification, or notification APIs and verify rate limiting and abuse controls.

  • Replay apply-coupon, referral, wallet-credit, reward-claim, or benefit APIs and verify one-time/limit enforcement.

  • Modify price, quantity, discount, currency, paymentStatus, planId, subscriptionId, or invoiceId during payment-related API calls.

  • Replay successful transaction, order, booking, subscription, or reset requests and verify idempotency/replay protection.

  • Call final-step APIs directly without completing required previous steps, such as reset password without OTP or confirm order without payment.

  • Send parallel requests for one-time actions such as coupon redemption, wallet withdrawal, OTP verification, invite acceptance, or booking.

  • Use negative amount, zero quantity, huge quantity, invalid status, past/future dates, or unsupported enum values.

  • Check whether sensitive flows require CAPTCHA, device checks, rate limits, behavioral checks, or re-authentication where needed.

API7: Server-Side Request Forgery (SSRF) 8 items

  • Look for APIs accepting URL/webhook/link/imageUrl/avatarUrl/importUrl/callbackUrl/redirectUrl/feedUrl/pdfUrl.

  • Use Burp Collaborator/interactsh/ngrok URL to confirm whether the server makes outbound requests.

  • Try safe internal targets such as http://127.0.0.1, http://localhost, http://10.0.0.1, http://172.16.0.1, http://192.168.1.1.

  • Check whether cloud metadata URLs such as http://169.254.169.254 are blocked.

  • Try file://, ftp://, gopher://, dict:// where applicable. Only required safe protocols should be allowed.

  • Provide an allowed external URL that redirects to internal IP and verify final destination is revalidated.

  • Try encoded IPs, decimal/octal IPs, IPv6 localhost, mixed case hostnames, and DNS rebinding only in authorized testing.

  • Verify internal response content, error messages, service banners, or timing differences are not returned to the user.

API8: Security Misconfiguration 9 items

  • Verify Swagger/OpenAPI docs are not publicly exposing sensitive/admin/internal APIs in production.

  • Send malformed JSON, invalid types, missing required fields, and invalid tokens; responses must not expose stack traces or internal paths.

  • Send Origin headers from untrusted domains and verify sensitive APIs do not allow unsafe origins with credentials.

  • Use OPTIONS and method tampering to ensure unnecessary PUT/DELETE/TRACE methods are disabled or protected.

  • Search for /health, /actuator, /metrics, /debug, /env, /config, /server-status, /phpinfo, /test, /sample endpoints.

  • Verify HSTS, X-Content-Type-Options, CSP/frame protections, Referrer-Policy, and secure cookie flags where applicable.

  • Review headers and error responses for server versions, framework versions, internal hostnames, and environment names.

  • APIs must redirect or block HTTP and never transmit tokens, credentials, or PII over unencrypted channels.

  • Verify dev/stage/UAT APIs are not publicly exposed with weak controls or production data.

API9: Improper Inventory Management 8 items

  • Search for /v1, /v2, /v3, /old, /legacy, /beta, /deprecated endpoints and verify they are protected or removed.

  • Test whether old API versions have weaker auth, missing MFA, no rate limit, extra fields, or broken authorization.

  • Inspect JS/mobile traffic for APIs not documented in Swagger/OpenAPI or not known to the team.

  • Check test, dev, UAT, QA, staging, preprod, sandbox API domains and verify no public sensitive exposure.

  • Compare documented endpoints with actual traffic; undocumented sensitive APIs should be reviewed and protected.

  • Verify removed UI features do not still have active backend APIs.

  • Look for multiple API hosts such as api, admin-api, internal-api, mobile-api, partner-api and verify consistent controls.

  • Review frontend JS and source maps for hidden endpoints, old domains, internal API hosts, and hardcoded API keys.

API10: Unsafe Consumption of APIs 8 items

  • For payment, OAuth, SMS, email, webhook, or partner callbacks, verify signature validation and source verification.

  • Replay captured webhook/callback payloads and verify timestamp, nonce, and signature prevent reuse.

  • Modify paymentStatus, amount, orderId, userId, planId, subscriptionId, or provider response fields before callback processing.

  • Simulate third-party timeout/error/invalid response and verify the application does not mark payment/login/verification as successful.

  • Verify the app validates and sanitizes third-party API responses before using them in decisions or rendering.

  • Verify email_verified, subject ID, issuer, audience, and provider identity are validated before account login/linking.

  • If users can register webhook URLs, verify SSRF protections, ownership checks, and safe outbound requests.

  • Confirm APIs do not send unnecessary PII, tokens, internal IDs, or secrets to third-party services.

Common API Functionality Checks 10 items

  • Test brute force, account enumeration, rate limiting, invalid credentials, malformed JSON, token leakage, and response consistency.

  • Test account enumeration, rate limiting, reset token leakage in response, host header poisoning, email parameter pollution, and replay.

  • Test token expiry, token reuse, token tampering, weak password acceptance, old password reuse, race condition, and session invalidation.

  • Test OTP brute force, expiry, reuse, resend limits, null/000000 values, code leakage in response, and verification for wrong user/session.

  • Test mass assignment, stored XSS, email/phone verification bypass, unauthorized field update, and CSRF if cookie-based.

  • Test file type, double extension, MIME mismatch, magic bytes, SVG/HTML XSS, large file, unauthorized upload, and direct file access.

  • Test IDOR on file IDs, path traversal, excessive export, PII leakage, expired signed URLs, and access after logout.

  • Test injection, excessive data exposure, pagination abuse, unauthorized filters, and performance/resource limits.

  • Test low-privilege access, forced browsing, BFLA, role tampering, bulk action abuse, and method-based bypass.

  • Test signature validation, replay protection, callback tampering, SSRF through callback URLs, and fail-open behavior.

Quick API Payload / Manipulation Reference 10 items

  • Change IDs: /users/1001 → /users/1002, ?user_id=1002, {"userId":1002}, X-User-ID: 1002.

  • Add fields: isAdmin=true, role=admin, status=active, verified=true, plan=premium, balance=99999, permissions=[admin].

  • Decode JWT, change role/user_id/tenant_id/exp, try alg=none only in authorized testing, then replay.

  • Send Origin: https://evil.example and check Access-Control-Allow-Origin and Access-Control-Allow-Credentials behavior.

  • Try URL fields with http://127.0.0.1, http://localhost, http://169.254.169.254, and Burp Collaborator URL.

  • Send duplicate parameters: ?id=1&id=2 or email=victim@test.com&email=attacker@test.com.

  • Try changing IP headers (X-Forwarded-For), user-agent, device ID, email case, phone format, endpoint version, or account rotation.

  • Try X-HTTP-Method-Override: DELETE, X-Method-Override: PUT, or _method=PATCH on restricted endpoints.

  • Try introspection query, deep nesting, batching, unauthorized fields, and IDOR in object arguments.

  • Send malformed JSON, wrong data types, missing fields, invalid enum values, and oversized payloads to check for stack traces.
